Employee Payroll System

Demonstrating the Four Pillars of Object-Oriented Programming in Java

1
Encapsulation

In Employee.java, the fields name, id, and baseSalary are all private. No code outside the class can touch them directly. They are read through getters, and baseSalary can only be changed through setBaseSalary(), which rejects any negative value before allowing the change.

Employee.java
private String name;
private String id;
private double baseSalary;   // hidden — cannot be set directly

public String getName()       { return name; }
public double getBaseSalary() { return baseSalary; }

public void setBaseSalary(double s) {
    if (s < 0) return;   // rejects negative salary
    baseSalary = s;
}
Self-Check
  • ✔ baseSalary is private — nobody can write employee.baseSalary = -500
  • ✔ getBaseSalary() is the getter
  • ✔ setBaseSalary() is the controlled setter — it rejects negative values
2
Inheritance

Manager and Cashier both extend Employee. They automatically inherit name, id, baseSalary, and all the getters — without rewriting any of it. Each child then adds its own extra fields on top: Manager adds a bonus, and Cashier adds overtimeHours and overtimeRate.

Manager.java
class Manager extends Employee {

    private double bonus;

    public Manager(String name, String id, double baseSalary, double bonus) {
        super(name, id, baseSalary);   // calls Employee's constructor
        this.bonus = bonus;
    }
    ...
}
Cashier.java
class Cashier extends Employee {

    private int overtimeHours;
    private double overtimeRate;

    public Cashier(String name, String id, double baseSalary, int overtimeHours, double overtimeRate) {
        super(name, id, baseSalary);   // calls Employee's constructor
        this.overtimeHours = overtimeHours;
        this.overtimeRate  = overtimeRate;
    }
    ...
}
Self-Check
  • ✔ Employee is the parent class
  • ✔ Manager is child class 1 — adds a bonus
  • ✔ Cashier is child class 2 — adds overtime hours and rate
  • ✔ Both use super() to pass shared data up to the parent constructor
3
Polymorphism

Both children override calculatePay() using @Override. The logic is different in each: a Manager's pay is base salary plus a fixed bonus, while a Cashier's pay is base salary plus overtime hours multiplied by the overtime rate. When the same calculatePay() call is made through an Employee variable, Java automatically runs the correct version for the actual object.

Manager.java — base + fixed bonus
@Override
public double calculatePay() {
    return getBaseSalary() + bonus;
    // e.g. 1,500,000 + 500,000 = TZS 2,000,000
}
Cashier.java — base + overtime
@Override
public double calculatePay() {
    return getBaseSalary() + (overtimeHours * overtimeRate);
    // e.g. 800,000 + (20 × 5,000) = TZS 900,000
}
Main.java — same call, different result
Employee e1 = new Manager("Julius", "MGR-001", 1500000, 500000);
Employee e2 = new Cashier("Asha",   "CSH-001", 800000,  20, 5000);

e1.calculatePay();   // → TZS 2,000,000  (Manager's version)
e2.calculatePay();   // → TZS 900,000    (Cashier's version)
Self-Check
  • ✔ calculatePay() is overridden in both child classes with @Override
  • ✔ Same method call produces a different result depending on the object
  • ✔ getRole() is also overridden in both children
4
Abstraction

Employee is declared abstract, so you cannot create a plain Employee object — it is only a blueprint. The two abstract methods calculatePay() and getRole() have no body in the parent; they are just a promise that every child class must fill in its own version.

Employee.java
public abstract class Employee {
    ...
    public abstract double calculatePay();  // no body — child must implement
    public abstract String getRole();       // no body — child must implement
}
This causes a compile error — you cannot build the blueprint directly
Employee e = new Employee(...);   // ERROR — abstract class cannot be instantiated
Employee e = new Manager(...);    // OK — use a concrete child class instead
Self-Check
  • ✔ Employee is declared abstract
  • ✔ calculatePay() and getRole() are abstract — no body, just a promise
  • ✔ Both Manager and Cashier are forced to implement them
